## Read-only git access layer

Added src/repo_history/git.py, a GitRepo class that shells out to git via subprocess argument lists to walk commits chronologically, report per-file churn with rename detection, fetch raw diffs, and resolve tags. It is paired with a synthetic fixture repo (tests/conftest.py) containing a rename, an extension, a revert, and an annotated tag, so the layer is exercised against realistic history shapes.

**Architecture:** Establishes the data-source boundary for the whole tool: everything downstream consumes frozen dataclasses (Commit, FileChange, Tag) rather than raw git output, and the module is deliberately read-only — no code path writes to the analyzed repository. Diffs are not carried on Commit; they are fetched on demand via raw_diff(sha), keeping history walks cheap.

## The full plan → skill → build pipeline

Built the whole repo-history engine in five steps: a pluggable analysis framework with a deterministic `mechanical` method, materialization of episodes into secret-scrubbed Markdown bundles, Pydantic-validated artifact rendering via `build`, the packaged /repo-history Claude Code skill that does the LLM map/reduce, and incremental state tracking so re-runs only analyze new commits. The design splits the work into two deterministic halves (plan, build) with the LLM sandwiched between them, communicating only through on-disk JSON.

**Architecture:** Introduced the pipeline's backbone: `analysis/` (base framework + registry + `mechanical` method + method-agnostic util), `security.py` (secret scrubbing), `work.py` (materialize episodes into `.repo-memory/.work/`), `models.py` (Pydantic contracts for LLM output), `build.py` (render durable artifacts), `state.py` (incremental head tracking), and `skill/SKILL.md` + `skillfile.py` (the packaged Claude Code skill). Every stage talks to the next only through files on disk, so the deterministic CLI and the LLM agent stay decoupled.
